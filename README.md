## 💡 Ключевые кейсы из практики

### Кейс 1. Ужесточение валидации полей авторизации и регистрации
* **Контекст:** Система пропускала некорректные данные в критических полях формы регистрации и входа.
* **Мои действия:** Провел целенаправленное тестирование граничных значений. Выявил, что бэкенд успешно обрабатывает запросы с множественными пробелами в полях `login`, `password`, `email_code` и `phone_code`, а также допускает регистрацию с именем из одного символа.
* **Результат:** Оформил серию баг-репортов с примерами cURL-запросов. Разработчики внедрили строгую валидацию (trim, проверка длины и формата), что исключило создание «мусорных» записей в БД.

### Кейс 2. Локализация ошибки сериализации изображений на фронтенде
* **Контекст:** При сохранении карточки ЖК в админке бэкенд стабильно возвращал `422 Unprocessable Content` с ошибкой валидации по всем полям `images.*`.
* **Мои действия:** Проанализировал вкладку Network в Chrome DevTools. Обнаружил, что фронтенд (React) при формировании `multipart/form-data` некорректно сериализовал объекты изображений, отправляя на сервер строку `"[object Object]"` вместо бинарных данных.
* **Результат:** Предоставил разработчикам точный пример некорректного Payload и cURL. Проблема была локализована на стыке фронтенда и бэкенда и исправлена в рамках одного спринта.

### Кейс 3. Анализ рассинхронизации данных с Bitrix24
* **Контекст:** Задача в Celery завершалась со статусом `succeeded`, но контакт не создавался в CRM.
* **Мои действия:** Взял UUID задачи из описания тикета, поднял логи в Grafana/Loki. Обнаружил, что API Gateway отправлял обязательное поле `LAST_NAME` со значением `None`, из-за чего CRM возвращала `400 Bad Request` с сообщением `Text must be greater not empty string`.
* **Результат:** Сформировал баг-репорт с выдержкой из лога Celery и указанием на конкретное поле контракта, что позволило бэкенд-разработчику исправить маппинг данных за несколько часов.

### Кейс 4. Тестирование валидации пагинации API
* **Контекст:** Эндпоинт каталога `GET /api/v2/estate/housing_complexes` не валидировал параметр `page[size]`.
* **Мои действия:** Отправил запрос с отрицательным значением `page[size]=-337`. Сервер вернул `200 OK`, а в `meta.per_page` пришло отрицательное число.
* **Результат:** Задокументировал отсутствие валидации. Была внесена рекомендация и реализовано правило: `page[size]` должен быть целым числом в диапазоне `1 ≤ N ≤ max`, иначе возвращается `400 Bad Request`.

### Кейс 5. Проверка матрицы доступов в White-Label системе
* **Контекст:** Необходимо было убедиться в корректной изоляции данных между франшизами (Пенза / Москва).
* **Мои действия:** Протестировал сценарии межрегиональных заявок согласно ТЗ. Проверил, что агент из одного региона не может редактировать финансовые атрибуты сделки другого региона. Убедился, что при клонировании контура бизнес-данные (сделки, клиенты) не утекают, а глобальные справочники соответствуют эталону.
* **Результат:** Подтвердил соответствие фактического поведения системы заявленным требованиям безопасности и изоляции данных.

## 📄 Пример артефакта: Как я оформляю Bug Report

Ниже — пример структурированного баг-репорта, демонстрирующий мой подход к локализации проблем.

**Заголовок:** `[API] Исключение FileDoesNotExist при обработке смешанного payload (загрузка + reorder)`  
**Окружение:** Dev / Stage  
**Эндпоинт:** `POST /api/v1/admin/estate/housing_complexes/{id}`  

**Описание проблемы:**  
При обработке multipart-запроса, который одновременно содержит данные для загрузки нового изображения и метаданные для изменения порядка (reorder) уже существующих, процесс завершается фатальной ошибкой 500. Фронтенд корректно формирует запрос, файл доставляется на сервер, однако на этапе сохранения возникает исключение.

**Фактическое поведение:**  
Запрос завершается ошибкой:  
`Spatie\MediaLibrary\MediaCollections\Exceptions\FileDoesNotExist: File /tmp/phperEklD does not exist.`

**Шаги воспроизведения:**
1. Авторизоваться в админке.
2. Перейти в редактирование ЖК.
3. Добавить новое изображение и изменить порядок существующих.
4. Нажать «Сохранить» и открыть DevTools → Network.

**Ожидаемое поведение:**  
Система должна дифференцировать элементы массива `images`: новые файлы обрабатывать методом `addMedia()`, а для существующих объектов (определяемых по наличию `id`) обновлять только метаданные (порядок) без попыток повторной физической загрузки из `/tmp`.
