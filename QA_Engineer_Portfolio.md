💼 Ключевые инженерные кейсы из практики
Кейс 1. Контрактное тестирование и защита от Race Condition в API регистрации
Контекст: Тестирование многошагового флоу регистрации с предварительной проверкой телефона (POST /check-phone) и финальным созданием пользователя (POST /register). Бизнес-требование: система должна различать свободные номера, дубли и «незавершенные» регистрации, обновляя данные последних без ошибок.
Мой вклад и действия:
Составил матрицу переходов состояний и проверил граничные значения: пустые строки, строки из пробелов после trim(), невалидные форматы.
Выявил уязвимость: эндпоинт check-phone возвращал 200 OK для строк, состоящих только из пробелов, вместо ожидаемого 422 Unprocessable Entity.
Протестировал сценарий Race Condition: смоделировал ситуацию, когда состояние БД меняется между вызовом check-phone и POST /register. Убедился, что бэкенд сохраняет строгую валидацию (Defence in depth) и не позволяет создать дубль, даже если клиент обошел pre-check.
Проверил нормализацию данных на фронтенде и бэкенде: автоматическая замена ведущих 8 и +8 на +7 должна применяться идемпотентно до отправки обоих запросов.
Результат: Предотвращен вывод на прод багов, приводящих к засорению БД дублями пользователей и некорректной работой CRM. Документировал контракт: phone_status ∈ {free, unregistered_continue, registered_duplicate} с соответствующими HTTP-кодами.
Кейс 2. Локализация скрытых ошибок синхронизации в микросервисной архитектуре
Контекст: Двусторонняя синхронизация контактов между веб-платформой, API Gateway и CRM. Данные уходили с сайта, задача в Celery завершалась со статусом succeeded, но сущность в CRM не создавалась.
Мой вклад и действия:
Вместо эскалации «баг в синхронизации», я провел трассировку запроса через систему мониторинга (Loki/Grafana) по UUID задачи.
Обнаружил, что API Gateway отправлял обязательное поле LAST_NAME со значением None (null), в то время как CRM требовала непустую строку (Text must be greater not empty string), возвращая 400 Bad Request.
В другом инциденте выявил проблему маппинга полей: UUID клиента записывался в кастомное поле UF_CRM_1716..., а сделка пыталась привязать его через поле UF_CRM_45_1715..., из-за чего CRM не находила клиента.
Проверил сценарий: при отсутствии обязательного поля agency_uuid в запросе к внешнему API CRM возвращала 422 Unprocessable Entity, что прерывало бизнес-процесс создания юридической сущности.
Результат: Сокращено время на дебаг для бэкенд-разработчиков. Мои баг-репорты с готовыми выдержками из логов Celery и указанием на конкретные поля контракта позволяли исправлять интеграционные ошибки за часы, а не дни.
Кейс 3. Отладка сетевых запросов и сериализации данных на фронтенде
Контекст: При сохранении карточки объекта недвижимости через админку (POST /api/v1/admin/estate/housing_complexes/{id}) бэкенд стабильно возвращал 422 Unprocessable Content с ошибками валидации по всем полям images.*.
Мой вклад и действия:
Проанализировал вкладку Network в Chrome DevTools и тело запроса multipart/form-data.
Обнаружил, что фронтенд (React) некорректно сериализовал объекты изображений, отправляя на сервер строку [object Object] вместо бинарных данных или корректных JSON-метаданных.
Выявил смежную ошибку: при попытке бэкенда (Laravel + Spatie MediaLibrary) обработать смешанный payload (загрузка нового файла + смена порядка существующих) возникало фатальное исключение FileDoesNotExist: File /tmp/phpXXXXXX does not exist, так как сервис пытался физически загрузить уже существующие ID.
Предложил решение: дифференцировать массив images на фронте (новые файлы vs объекты с id), а на бэкенде — проверять наличие id перед обращением к временной директории /tmp.
Результат: Локализовал проблему на стыке фронтенда и бэкенда. Баг-репорт с точным указанием на ошибку сериализации и примером некорректного Payload позволил разработчикам исправить проблему за один спринт.
Кейс 4. Тестирование сложной фильтрации в Elasticsearch
Контекст: В каталоге недвижимости при комбинации фильтров «Комнатность: все» и «Площадь: до 1000 м²» из выдачи бесследно исчезали жилые комплексы.
Мой вклад и действия:
Воспроизвел проблему, сформировав точный JSON-запрос, уходящий на бэкенд: filter[rooms][]=0,1,2,3,4 и filter[apartment_total_area_to]=1000.
Проанализировал логику построения запроса к Elasticsearch: условие minimum_should_match некорректно обрабатывало пустые массивы, которые генерировались на бэкенде при выборе опции «все комнатности».
Проверил сценарий fallback: при недоступности Elasticsearch система должна переключаться на Materialized Views (housing_complex_catalog_mv), и фильтрация должна работать аналогично.
Результат: Дефект был исправлен на уровне сервиса ElasticHCService. Подтвердил, что после фикса пагинация, сортировка и fallback-сценарии работают корректно.
Кейс 5. Тестирование матрицы доступов в White-Label (франшизной) системе
Контекст: Платформа поддерживает множество независимых франшиз (контуров). Необходимо было протестировать правила межрегионального доступа к сделкам.
Мой вклад и действия:
Протестировал матрицу условий (Поставщик × Агент). Например, агент из одного региона не должен иметь права редактировать финансовые атрибуты сделки, если объект принадлежит другому региону.
Выявил уязвимость: правило view (запрет на просмотр) применялось только через фильтрацию списков, но при переходе по прямой ссылке (Deep Link) карточка сделки открывалась без ограничений.
Проверил сценарии изоляции данных при клонировании контура: убедился, что в новой среде отсутствуют утечки бизнес-данных (ЖК, клиенты, сделки) из других франшиз, а глобальные справочники совпадают с эталоном.
Протестировал конфигурацию white-label: фронтенд получает конфигурацию (логотипы, цвета, настройки меню) от бэкенда в зависимости от франшизы пользователя. Проверил, что при смене роли (верификация агента) происходит инвалидация кэша конфигурации и получение обновленной версии.
Результат: Предотвращены критические уязвимости безопасности и утечки данных между конкурентирующими франшизами внутри одной экосистемы.
Кейс 6. Тестирование валидации пагинации и граничных значений
Контекст: Эндпоинт каталога (GET /api/v2/estate/housing_complexes) не валидировал параметр пагинации page[size].
Мой вклад и действия:
Передал отрицательное значение: page[size]=-337.
Запрос вернул 200 OK, а в meta.per_page пришло переданное отрицательное число.
Проверил также нулевые и сверхбольшие значения.
Задокументировал отсутствие валидации на бэкенде.
Результат: Предложил правило: page[size] должен быть целым числом в диапазоне 1 ≤ page[size] ≤ max (например, 100), при невалидном значении — возврат 400 Bad Request или 422 Unprocessable Entity.
Кейс 7. Анализ инцидента через систему мониторинга
Контекст: При создании застройщика в CRM данные не попадали в БД сайта. В логах фиксировалась ошибка permission denied for table developers.
Мой вклад и действия:
Открыл систему мониторинга (Grafana/Loki).
Нашел логи фонового воркера по UUID задачи.
Обнаружил, что у пользователя БД, под которым работает воркер, отсутствуют права на запись в таблицу developers.
Результат: Локализовал проблему на уровне прав доступа к БД. После предоставления прав проверил, что данные успешно синхронизируются.

Ниже — пример структурированного баг-репорта, демонстрирующий мой подход к локализации проблем на уровне HTTP и кода.
Заголовок: [API] Исключение FileDoesNotExist при обработке смешанного payload (загрузка + reorder)
Окружение: Dev / Stage
Эндпоинт: POST /api/v1/admin/estate/housing_complexes/{id}
Описание проблемы:
При обработке multipart-запроса, который одновременно содержит данные для загрузки нового изображения и метаданные для изменения порядка (reorder) уже существующих, процесс завершается фатальной ошибкой 500. Фронтенд корректно формирует запрос, файл доставляется на сервер, однако на этапе сохранения возникает исключение.
Фактическое поведение (с анализом):
Запрос завершается ошибкой:
Spatie\MediaLibrary\MediaCollections\Exceptions\FileDoesNotExist: File /tmp/phperEklD does not exist.
Анализ: Бэкенд не дифференцирует элементы массива images. Он пытается применить метод физического сохранения (addMedia()) ко всем элементам массива, включая те, у которых уже есть id и которые требуют только обновления метаданных (order_column).
Ожидаемое поведение:
Система должна проверять наличие id в элементе массива. Если id передан — обновлять только метаданные без попыток чтения файла из временной директории /tmp.
